# レイヤードフロント（HTML / CSS / JS）

**構造・見た目・振る舞い**をファイルと責務で分離し、保守しやすいバニラ（または軽量）フロントを組み立てる。

## 言語

- **ユーザーとのやり取りは原則すべて日本語**とする。
- ユーザーが明示的に別言語を求めた場合のみ切り替える。

## 基本原則

| レイヤー | 担当 | やらないこと |
|----------|------|----------------|
| **HTML** | 文書構造・意味・コンテンツの骨格・アクセシビリティの土台 | 見た目の細部（色・余白・配置）、複雑な動作ロジック |
| **CSS** | レイアウト・タイポ・色・余白・アニメーション・レスポンシブ・状態の見た目（`:hover` 等） | ビジネスロジック・データ取得・DOM の意味の変更 |
| **JS** | インタラクション・状態更新・イベント・（必要なら）軽い DOM 操作・ API 呼び出し | 可能な限りマークアップやスタイルの詳細を直書きしない |

**ルールオブサム**: 変更理由が「意味の変化」→ HTML、「見え方」→ CSS、「ふるまい」→ JS。

## HTML の方針

- **意味のある要素**を使う（`header` / `main` / `nav` / `section` / `button` / `label`+`input` 等）。装飾専用の `div` 乱用は避ける。
- 操作できるものは **`button`** または **`a`**。`div` + `click` は避ける（どうしてもなら理由と a11y 対策）。
- JS との接点は **`id`（単一）**、**`class`（スタイル・BEM 等）**、**`data-*`（スクリプト用フック）** に集約。JS からのフックは `js-` や `data-ui-*` など**役割が分かる接頭辞**を検討。
- **インライン `style`** は原則禁止（動的値や例外のみ最小限）。
- **`onclick` やインラインスクリプト`** は避け、別ファイルの JS にイベントを寄せる。

## CSS の方針

- **グローバルなトークン**（色・フォント・スペーシング・角丸・影）を `:root` または先頭セクションにまとめ、プロパティは **`var(--token)`** で参照。
- **レイアウト**（Flex / Grid）と **タイポ・カラー** と **コンポーネント単位のブロック**を、ファイルまたはコメントで区切り、読みやすく並べる。
- **状態**は CSS で表現できるものは CSS に（`:hover`、`:focus-visible`、`[aria-expanded="true"]`、`.is-open` など）。JS はクラスや属性の付け替えにとどめ、ピクセル単位の見た目計算は避ける。
- **レスポンシブ**はメディアクエリやコンテナクエリで CSS に閉じる。JS で画面幅分岐を乱用しない（必要なときだけ）。
- **命名**: プロジェクトで一貫（BEM、ユーティリティ中心、など）。とにかく**一貫性**を優先。

## JS の方針

- **DOM のクエリとイベント登録**をエントリ（例: `DOMContentLoaded`）にまとめ、**機能ごとに小さな関数**へ分割。
- **HTML 文字列のテンプレート大量連結**より、`createElement` / `DocumentFragment`、または最小限のテンプレートリテラルで**構造が読める**ようにする。
- **状態**は変数や小さなモジュールに保持し、UI 更新は「状態 →（必要なら）クラス/属性 → CSS が反応」の流れを優先。
- **fetch や API** は JS に置く。取得結果の「表示の変化」はクラス付け替えと CSS で済ませられることが多い。
- **バンドラなし**でも動くように、**グローパル汚染を避け**（IIFE、`type="module"`、単一エントリ）意識する。

## ファイル構成（目安）

デフォルトは**役割でファイルを分ける**:

```
project/
index.html # マークアップのみ（link/script で外部参照）
css/
tokens.css # 任意: 変数のみ
base.css # リセット・タイポの土台
layout.css # ヘッダー/メイン/フッター等
components.css # ボタン・カード等（規模に応じて分割）
js/
main.js # エントリ（初期化）
ui/<feature>.js # 規模が出たら機能単位で分割（任意）
```

- HTML には **`link rel="stylesheet"`** と **`script defer`（または末尾 `type="module"`）** を用意。スタイルとスクリプトは**別ファイル**。
- 1 画面が小さい場合は `styles.css` / `main.js` の 2 ファイルでもよい。**分け方は「検索しやすさ」優先**。

## 実装の順序（推奨）

1. **HTML だけ**で情報の順序と見出し階層を正す（無スタイルでも読める）。
2. **CSS** でレイアウトとトークンを当てる（レスポンシブ含む）。
3. **JS** でインタラクションを足す（付けすぎない）。

## アンチパターン

- HTML にビジネスロジックや長いスタイルブロックを詰め込む。
- JS で `element.style.left = ...` を多用してレイアウトを組む（CSS に任せる）。
- 同じ見た目を HTML のクラスと JS のスタイル両方で管理する（単一の情報源に）。
- 「とりあえず」全て `app.js` に書く（数百行超える前に分割）。

## 他スキルとの関係

- **見た目の質・品格を高めたい**場合は、`distinctive-frontend` と併用してよい（本スキルは**責務分離とファイル構成**が主題）。

## 納品前チェック

- [ ] 見た目・レイアウトの変更は主に CSS で完結するか
- [ ] 動き・データ・イベントは JS に閉じ、HTML はフックだけか
- [ ] インラインスタイル・インラインスクリプトが最小か
- [ ] `button` / `label` / 見出しレベルなど、a11y の基本が HTML で担保されているか
